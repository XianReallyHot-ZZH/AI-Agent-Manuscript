一句话先讲懂：模型在一个环境里循环工作

Agent 不是某个神秘类，而是一个不断重复的闭环：看见上下文，决定下一步，调用外部能力，再把结果带回上下文。

观察

用户消息、历史对话、文件、工具结果、系统状态。

判断

模型选择直接回答，还是请求某个工具。

行动

Harness 执行工具，真正改变外部世界。

回写

工具结果作为新消息进入上下文。

继续

直到模型给出结束信号或系统中断。

三个角色，比“框架名词”更重要

Model

负责选择。它读上下文，决定回答、调用工具、继续探索或停止。

Harness

负责边界。它管理消息、工具、权限、重试、压缩和持久化。

World

负责事实。文件系统、Shell、渠道、数据库、MCP 服务都在这里。

User

提出目标、约束和反馈。

Harness

组织状态、策略、工具和上下文。

Model

根据上下文做选择。

World

文件、Shell、渠道、数据库和外部服务。

Back

结果回到 Harness，再进入下一轮。

强调模型和 Harness 的分工。模型不直接碰外部世界，所有副作用都经过 Harness。

Part 2 / Core Architecture

Agent 的本质：可行动的循环

AI-Agent 说 The Model IS the Agent；工程上要补一句：Harness 决定这个 Agent 能不能安全地行动。

第二部分从本质、边界、协议三个层面讲清楚 Agent。

“The Model IS the Agent”不是说代码不重要

模型提供智能

理解目标、拆解任务、选择工具、判断是否继续，这些主要发生在模型里。

代码提供世界

代码负责把外部世界变成模型能读懂、能调用、能被约束的接口。

好的 Agent 工程不是把规则写死。它是把上下文、工具和边界组织好，让模型的选择能落到真实环境里。

最小 Loop 怎么长成系统？看四个扩展点

Build Context

加入历史、计划、技能摘要、记忆检索和运行时约束。

Call Model

保持协议小：模型只需要表达回答、工具请求或停止。

Tool Boundary

Schema、Permission、Hook、Handler 都发生在副作用之前。

Result Re-entry

工具结果、错误和恢复信息回写，成为下一轮事实。

横向扩展
增加工具、渠道、MCP、子 Agent，但都不改变 Loop 的基本节奏。
纵向增强
在上下文、权限、恢复、审计上加层，让每一轮更可控、更可恢复。
这张图解释“为什么最小代码不等于简单系统”。Loop 本身短，是因为复杂度被放到了四个明确扩展点上。
用这页把 5 行 agentLoop 和后面几十个机制连接起来，避免听众觉得后续内容是突然冒出来的功能清单。

stop_reason 是模型和 Harness 的最小协议

end_turn

模型认为当前任务可以交付，Harness 把文本返回给用户。

tool_use

模型需要外部事实或副作用，Harness 执行工具并回写结果。

error / limit

真实系统还要处理异常、限流、上下文溢出、超时和取消。

协议小，系统才稳。模型不需要知道工具内部实现，只需要表达“我要什么能力”。

系统做

Harness 把这些信息组织成上下文，模型选择 read_file 或 bash，工具结果再被回写为下一轮事实。

Goal

用户目标进入 messages。

Inspect

Tool 不是函数列表，而是控制面 + 执行面

控制面：Schema

告诉模型有什么工具、什么时候用、输入长什么样。它影响模型的选择。

执行面：Handler

真正读文件、写文件、跑 Shell、发消息。它影响真实世界的状态。

Tool Schema

面向模型的契约：名称、描述、输入结构。

Model Choice

模型输出 tool_use，表达想调用什么能力。

Policy Gate

allow、deny、ask，把副作用关进边界。

Handler

真正执行文件、Shell、渠道或 MCP 操作。

tool_result

把事实和错误反馈送回上下文。

为什么 Tool 要拆成 Schema 和 Handler？

如果混在一起拆开以后架构收益
模型看不懂工具边界Schema 面向模型描述能力模型更容易选对工具
权限只能散落在函数里Policy Gate 统一拦截 tool_use危险操作有统一入口
工具实现难以替换Handler 面向系统执行副作用教学版 Map、生产版注入都能承载
错误反馈不稳定tool_result 统一回到上下文模型能基于同一种反馈继续推理
这不是过度设计。只要工具会改变外部世界，控制面和执行面分离就是最小安全边界。

一次 Tool 调用，其实经过四道门

1. 让模型看懂

Schema 描述工具名、能力边界、输入结构和使用时机。

->
2. 让系统判断

Policy 根据工具、路径、命令、用户和上下文决定 allow、ask、deny。

->
3. 让世界执行

Handler 读写文件、运行 Shell、调用 MCP 或投递外部消息。

->
4. 让模型继续

tool_result 把事实、错误、权限拒绝和恢复建议回写上下文。

Tool 的关键不是“能不能调函数”。关键是每次副作用都有契约、策略、执行和反馈四个位置可以治理。
这页把 Tool 的 Schema、Policy、Handler、tool_result 串成一条治理链，帮助听众理解为什么后面会出现权限、Hook、MCP。

Context 是 Agent 的工作记忆，不只是聊天记录

目标

用户到底要完成什么。

过程

已经尝试过什么，失败在哪里。

事实

文件、工具输出、环境状态。

约束

权限、风格、系统提示词。

Agent 失忆通常不是模型突然变笨。更多时候是上下文组织、压缩、持久化和回写策略出了问题。

问题模型负责Harness 负责

Part 3 / From Loop To Usable

让 Agent 可用：管理信息，而不是堆功能

规划、技能、子 Agent、压缩、多渠道，本质上都在解决一个问题：模型这轮到底应该看见什么？

第三部分重心从“有什么功能”改为“这些功能在管理什么信息”。

最小 Loop 很快会遇到四类失败

遗忘

历史太长，关键约束被挤掉。

跑偏

任务没有外化计划，模型反复换方向。

过载

把所有知识一次塞进 prompt。

失控

工具副作用没有权限、审计和恢复。

所以项目课程不是在堆 feature。它们是在逐步补齐一个能长期工作的运行系统。
这页解释为什么从 S03 开始会出现 Todo、Skill、Context Compact 等机制。

可用智能体的三角：计划、知识、上下文

计划

让模型知道“现在做到哪一步”，避免在长任务中反复换方向。

知识

让模型知道“有什么可用知识”，但只在需要时加载细节。

上下文

让模型看到“当前最重要事实”，并在窗口限制下继续工作。

TodoWrite

把进度外化。

Skill

把知识分层。

Subagent

把探索隔离。

Compact

把历史重写。

Channel

把输入标准化。

可用性的核心不是加更多工具。它是持续降低模型每一轮的认知负担。
用三角模型承接 Part 3 后续机制，让听众先知道这些机制之间的关系。

信息生命周期：不是塞 prompt，而是让信息流动

External State

->
Select

按当前目标筛出必要事实和知识目录。

->
Working Context

模型这一轮真正能看到的信息。

->

窗口压力出现时，把历史重写成交接摘要。

->
Recall

后续按任务重新检索、注入、继续工作。

Todo

把任务进度从隐式记忆变成外部状态。

Skill

先给目录，需要时再加载正文。

Subagent

把高噪声探索隔离，只带回摘要。

规划：把“我要做什么”从脑子里拿出来

TodoWrite

计划不是替模型思考，而是让模型把中间目标写成外部状态，后续每轮都能看见。

Nag Reminder

如果多轮没有更新计划，系统把提醒插回上下文，逼模型重新对齐任务状态。

核心原则：长期任务不要只靠模型隐式记忆，要把进度变成可检查的状态。

技能：知识要按需加载，不要一次塞满

Catalog

只注入技能名称和描述，让模型知道有什么。

Light Prompt

保持系统提示词轻，不把知识库一次塞满。

Need Detail

模型判断当前任务是否需要完整技能内容。

load_skill

按需读取完整 body。

Continue

带着必要知识继续执行。

两层知识注入的价值：先让模型知道“有什么”，只有需要时再给它“全部内容”。

子 Agent：开一个干净脑袋，带回摘要

独立上下文

子 Agent 不继承父 Agent 的全部消息历史，减少干扰。

专注任务

适合搜索、分析、验证这类边界清楚的子问题。

压缩返回

父 Agent 只拿回结果摘要，而不是整个探索过程。

这不是并发技巧。它首先是上下文隔离技巧。

强调“Process = Context Isolation”这一模式。

压缩：不是丢历史，而是重建继续工作的记忆

Raw History

历史很长，夹杂工具输出、失败尝试和临时推理。

Pressure

接近 token 上限，或 API 返回上下文溢出错误。

Summarize

保留目标、事实、决策、未完成工作和风险。

Rebuild

用摘要和当前状态重建下一轮上下文。

Continue

模型从压缩后的工作记忆继续执行。

好的 compact 不是一句“之前聊过很多”。它必须保留目标、已做决策、关键事实、未完成动作。

用户最终目标和硬约束必须保留这是继续工作的方向盘
已经验证过的事实保留结论和来源避免重复探索和误判
失败尝试和错误原因保留关键失败模式避免模型走回老路
大段工具输出和日志摘要化，只保留异常点节省窗口，降低噪声
中间闲聊和无效推理丢弃不帮助下一步行动
Compact 的产物应该像交接文档。另一个 Agent 只看摘要，也能继续把任务做完。
这页把压缩从抽象机制变成可判断的信息筛选标准。

这一层所有机制，都在回答同一个问题

机制解决的问题架构位置
TodoWrite任务进度不要只存在模型隐式记忆里Context 状态外化
Skill Loading知识太多，不能一次塞进 promptContext 按需注入
Subagent复杂探索会污染主线对话Context 隔离
Context Guard历史和工具结果会挤爆窗口Context 恢复
Channel平台事件格式不一致Input 标准化
主线不是“功能越来越多”。主线是让模型每一轮看到更正确、更少噪声、更可行动的信息。

Part 4 / Production Agent

生产级 Agent：先兜底，再变聪明

能跑 demo 不难；难的是工具有副作用、消息不能丢、上下文会溢出、请求会失败、多人会并发。

第四部分讲工程底线，少讲实现细节，多讲为什么生产系统要这样拆。

生产级底线：四件事必须可控

权限

危险工具必须能拦截、询问、审计。

恢复

上下文溢出、限流、超时要有路径。

投递

消息先落盘，再发给外部渠道。

并发

同一对象上的请求要排队或隔离。

Agent 越能行动，越需要工程刹车。没有这些底线，智能越强，事故半径越大。
这页是生产部分的总纲。

生产可靠性：把失败关进可恢复链路

Risky Intent

模型想写文件、跑命令、发消息或访问外部服务。

->
Guard

权限、Hook、输入校验先判断能不能做。

->
Durable State

关键操作先落盘，留下 pending、attempt 和 trace。

->
Execute

真正触达文件系统、渠道、模型 API 或外部服务。

->
Recover

失败时退避、压缩、重试、降级或人工升级。

生产系统不是追求“没有失败”，而是让失败发生在已知状态之间，系统能知道上一步做到哪里、下一步该怎么补偿。
可靠性链路把四件事串起来：权限控制事故入口，持久化保留恢复点，投递和恢复处理外部不确定性，并发控制状态污染。
这页把生产级底线合成一条失败治理链路，再展开权限、恢复、投递、并发就不会显得零散。

一次生产事故，通常是链式发生的

Tool Risk

模型请求写文件、跑命令或发消息。

Permission

危险动作先被策略拦截或要求确认。

Execution

执行失败、超时、限流或返回超大结果。

Recovery

重试、压缩、降级或保留待投递状态。

Audit

日志和 Hook 让问题能被复盘。

生产级架构的意义：不是保证永不失败，而是保证失败有边界、有状态、有证据。
用事故链路解释权限、恢复、投递、审计为什么要一起出现。

真正执行工具副作用。

PostToolUse

记录日志、观察结果、做转换。

tool_result

把结果送回模型上下文。

决定能不能做。

Hook

在生命周期里插入扩展。

审计

给生产排障留证据。

Part 5 / Multi-Agent And Gateway

多 Agent：复杂度来自边界，而不是数量

多开几个模型很容易；让它们可靠通信、分工、隔离和收敛才是难点。

第五部分把 AI-Agent Teams 和 OpenClaw Gateway 放在一起讲。

多 Agent 首先要回答四个工程问题

谁

身份、角色、权限和责任边界。

说什么

消息格式、状态、确认和失败语义。

改哪里

文件所有权、任务归属、目录隔离。

怎么停

完成、取消、交接、超时和恢复。

多 Agent 的核心不是“更聪明”。它首先是分布式协作问题。
这一页为邮箱、协议、worktree 做铺垫。

从单 Agent 到团队，要分清三条线

消息线

谁给谁发消息，消息是否可恢复，是否有 ack、timeout 和重试。

任务线

谁认领任务，任务状态在哪里，计划是否需要批准，失败如何交接。

代码线

谁能改哪些文件，并行修改如何隔离，什么时候合并回主线。

多 Agent 不只是通信。消息、任务、代码三条线任意一条没有边界，协作都会变成随机交错。
这页解释后面的 JSONL 邮箱、握手协议和 Worktree 是怎样互相补位的。

多 Agent 边界图：消息、任务、代码不能混在一起

Message Boundary

JSONL inbox、ack、timeout、shutdown 协议，解决“谁对谁说了什么”。

Task Boundary

任务认领、计划批准、状态交接，解决“谁负责做到什么程度”。

Code Boundary

worktree、文件所有权、审查合并，解决“谁能改哪里”。

Agent A
只直接修改自己的上下文、任务状态和隔离工作区；跨边界必须发消息或提交变更。
Agent B
通过协议接收任务和证据，不依赖对方的隐式记忆，也不直接共享可变状态。
多 Agent 的核心原则是少共享、强协议。能追加消息就不要共享内存，能隔离目录就不要同时改主工作区。
这页把 JSONL 邮箱、协议和 Worktree 三页放在同一张边界图里，降低听众对多 Agent 的抽象感。

JSONL 邮箱：最小可恢复通信协议

Sender

Agent 只追加消息，不直接改对方内存。

JSONL Inbox

每个 Agent 一个 inbox 文件，append-only、可审计、可恢复。

Receiver

接收方轮询 inbox，再更新自己的上下文和任务状态。

JSONL 的价值不是高级。它简单、可追加、可审计、崩溃后还能继续读。

来源：AI-Agent S15 Agent Teams

强调邮箱模型的可靠性和可解释性。

握手协议：协作需要状态机

Plan Approval

执行前先让对方确认计划，降低跑偏成本。

Shutdown

不是直接退出，而是发出意图、等待确认、交接状态。

Timeout

对方不响应时，系统要能继续前进或升级处理。

协议让协作可恢复。没有状态机，多 Agent 很快会变成互相丢消息。

来源：AI-Agent S16 Team Protocols

以协议而非代码讲 S16。

Worktree 隔离：并行工作先隔离目录

Task

任务进入共享队列。

Claim

Agent 认领工作，明确所有权。

Worktree

创建独立目录作为工作空间。

Edit

在隔离目录里修改和验证。

Review / Merge

审查后合并，再清理 worktree。

隔离不是洁癖。多个 Agent 同时改同一个仓库时，目录级隔离能显著降低冲突和误删风险。

来源：AI-Agent S18 Worktree Isolation

讲所有权和隔离，避免只讲 git 命令。

Agent Loop

Tool Router

统一决定走内置工具还是 MCP。

MCP Router

把工具调用翻译成协议消息。

JSON-RPC

通过 stdio 发送 JSON-RPC 2.0。

External Server

外部服务执行能力并返回结果。

MCP 的关键价值：外部工具仍然走同一个 schema、权限、执行、结果回写链路。

Appendix A3

几个容易混淆的词

词在这套架构里的含义
Harness围绕模型的运行环境：消息、工具、权限、恢复、持久化。
Tool模型能请求、系统能执行的外部能力，不等于普通函数。
Context模型这一轮能看到的信息集合，包括历史、工具结果、系统提示词。
Memory跨会话保存并按需召回的信息，不应该全量塞进 context。
Gateway把多渠道消息路由到正确 Agent，并负责投递、恢复和管理。
术语表页适合 Q&A 时回跳。
